# Local-First Delta Sync Architecture Plan

This document outlines the detailed architecture and implementation steps for migrating the application to an **Offline-First Local DB Architecture** using **Isar**. 

This approach completely resolves Firestore's read limits, solves the N+1 join problem, and enables infinite client-side searching and sorting at zero latency and zero recurring cost.

---

## 1. Core Architecture

Instead of the UI relying on unbounded Firestore streams (`.snapshots()`), the UI will rely entirely on a local **Isar database**. Firestore will only be used as a backend synchronization server.

### The Data Flow
1. **App Start**: App reads deposits and customers instantly from the local Isar database.
2. **Delta Sync**: A background `SyncService` queries Firestore for any documents where `updatedAt > last_sync_time`.
3. **Local Upsert**: The new/updated documents are saved into Isar. Isar's reactive streams (`.watch()`) automatically notify Riverpod to rebuild the UI.
4. **Writes**: When the user creates/updates a deposit, it is saved to Isar immediately (for instant UI feedback) and simultaneously pushed to Firestore.
5. **Soft Deletes**: Documents are no longer truly deleted in Firestore. They are marked with `isDeleted: true` and their `updatedAt` is bumped. The delta sync picks this up and deletes the local Isar record.

---

## 2. Implementation Steps & Order of Execution

To safely transition the app without breaking the existing Firestore implementation, the migration must be done in two distinct phases.

### Phase A: The Prerequisites (Data Safety First)
Before introducing a local database, we must ensure all new data generated by the app is "Isar-ready".

**Step 1: Base Model Updates**
*   Add `bool isDeleted = false;` to `Customer`, `RecurringDeposit`, and `OneTimeDeposit` models.
*   Update their `fromJson` factories to handle legacy documents by defaulting `isDeleted` to `false` if the field is missing.

**Step 2: Implement Soft Deletes in Repositories**
*   Rewrite the `deleteX()` methods in `CustomerRepository`, `RecurringDepositRepository`, and `OneTimeDepositRepository`.
*   Instead of calling `_collection.doc(id).delete()`, call `_collection.doc(id).update({'isDeleted': true, 'updatedAt': FieldValue.serverTimestamp()})`.
*   *Note: At the end of Phase A, the app still runs 100% on Firestore, but is now generating safe data.*

### Phase B: The Local Database Integration
Now that the data is safe, we introduce Isar to handle caching and reading.

**Step 3: Add Dependencies & Isar Models**
*   Add Isar and its generators to `pubspec.yaml`:
    ```yaml
    dependencies:
      isar: ^3.1.0
      isar_flutter_libs: ^3.1.0
      path_provider: ^2.1.2

    dev_dependencies:
      isar_generator: ^3.1.0
      build_runner: ^2.4.0
    ```
*   Add `@collection` to the domain models.
*   Add a local auto-incrementing Isar ID: `Id isarId = Isar.autoIncrement;`.
*   Ensure the Firestore string ID is indexed and unique: `@Index(unique: true, replace: true) String id;`.
*   Run `flutter pub run build_runner build`.

**Step 4: Implement the `SyncService`**
*   Create a dedicated `SyncService` (provided via Riverpod) that handles the delta logic for all collections.
*   **Logic:** Fetch `lastSyncTime` -> Query Firestore `updatedAt > lastSyncTime` -> Iterate snapshots -> If `isDeleted`, remove locally -> Else `isar.put()` -> Update `lastSyncTime`.

**Step 5: Refactor Repositories (Dual-Write)**
*   **Read**: Replace `.snapshots()` with `isar.collection.where().watch(fireImmediately: true)`.
*   **Create/Update/Delete**: Write to Isar first (instant UI update), then write/soft-delete to Firestore.

**Step 6: Refactor Riverpod Controllers**
*   Now that all data sits locally and updates instantly, Riverpod controllers (`recurring_deposits_controller.dart`, etc.) can load all records from Isar.
*   Because Isar handles thousands of records in memory effortlessly, your existing Dart logic (`filteredRecurringDeposits` using `.where()` and `.sort()`) will run instantly without costing a single Firestore read.

## 3. Advantages vs. Trade-offs

| Advantages | Trade-offs |
| :--- | :--- |
| **Zero Read Costs:** Read 10,000 documents instantly for free. | **Boilerplate:** Requires setting up Isar schemas and generators. |
| **Complex Queries:** Filter, sort, and substring search (`.contains()`) locally in Dart memory with no restrictions. | **Soft Deletes:** Real deletions in Firestore must be replaced with `isDeleted` flags to trigger delta syncs. |
| **N+1 Joins Solved:** Customers and Deposits exist locally; connecting them takes milliseconds. | **Sync Conflicts:** Assumes "Last Write Wins" based on Firestore timestamps. |
| **True Offline First:** UI never hangs waiting for network or Firebase cache checks. | |

## 4. Acceptance Criteria
1. The app starts and displays data instantly from the local Isar database.
2. The Firebase emulator shows exactly `0` reads if the app is reopened and no data has changed on the server.
3. Searching for a customer name in the deposits list is instantaneous and doesn't trigger server reads.
4. Deleting a deposit removes it locally and marks it as `isDeleted` on the server.
