# Session Log: 2026-06-02 (Session 4)

## Refactoring Connected Cards
- Identified that the separation between `OneTimeDepositCard` (Dumb) and `ConnectedOneTimeDepositCard` (Wrapper) was primarily done to allow drawing Skeleton loading states (which only take primitives) without triggering Riverpod actions, while maintaining Feature Isolation constraints.
- Optimized this architecture by deleting the wrapper pattern entirely.
- Created a single unified `OneTimeDepositCard` and `RecurringDepositCard` inside their respective feature presentation folders. These are now natively `ConsumerWidget`s that accept a `Deposit` model.
- Solved the skeleton loading problem by exploiting the existing `OneTimeDeposit.dummy` and `RecurringDeposit.dummy` factories. The unified cards dynamically detect if the ID is `'dummy'` and gracefully skip interaction callbacks and API requests if so, rendering perfectly as skeletons.

## Decisions Made
- Replaced the "Wrapper Component" pattern with a "Smart Component + Dummy States" pattern. This reduces file boilerplate, eliminates manual parameter-mapping (e.g., `amount: deposit.amount`), and maintains the rule that domains shouldn't leak to `core/`. We rely on Pragmatic Composition to share these smart cards with `CustomerDetailScreen`.
