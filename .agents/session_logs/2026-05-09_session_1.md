# Session Log: 2026-05-09

## Objective
Refactor Deposits (One Time Deposits and Recurring Deposits) to use Firestore repositories with real-time stream sync, similar to the Customers feature migration.

## Changes Made
- **Firestore Repositories**: 
  - Added `FirestoreOneTimeDepositRepository` to handle `one_time_deposits` collection.
  - Added `FirestoreRecurringDepositRepository` to handle `recurring_deposits` collection.
  - Both repositories implement a `watch*` stream to seamlessly sync Firestore snapshots with the UI.
  - Enabled creation of documents with UUID-based IDs explicitly supplied by controllers.
- **Repository Providers**: 
  - Updated `oneTimeDepositRepositoryProvider` and `recurringDepositRepositoryProvider` to conditionally return the `Firestore*Repository` or `Fake*Repository` based on the `USE_FAKE_DATA` environment variable.
- **Controllers**:
  - Updated `OneTimeDepositsController` and `RecurringDepositsController` to extend `Stream<UnmodifiableListView<T>>` rather than fetching futures.
  - Replaced future fetching with mapping the repository streams to `UnmodifiableListView` to enable real-time UI updates.
  - Removed explicit `ref.invalidateSelf()` from `save*` and `delete*` operations because the new stream architecture naturally pushes updates via Firestore.
  - Added UUID generation for `save*` methods directly in the controller if no `id` is provided, replacing the previous fake repo logic which generated IDs post-submission.
- **Generated Code**:
  - Ran `build_runner` to regenerate riverpod providers reflecting the `Stream` output of the controllers, resolving compiler errors.

## Decisions Taken
- Used client-side UUID generation for IDs inside controllers to maintain the same offline-persistence and optimistic UI pattern applied in the Customer feature.
- Retained the `Fake*Repository` implementation with an integrated `StreamController` to allow local fake-data testing.
