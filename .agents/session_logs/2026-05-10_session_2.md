# Session Log: 2026-05-10 (Session 2)

## Goal
Enforce Dumb Component Architecture for deposit cards and optimize Firestore reads.

## Changes Made
- **Data Layer**: 
  - Added `watchCustomerById(String id)` to `CustomerRepository` interface.
  - Implemented single-document stream fetching in `FirestoreCustomerRepository` and `FakeCustomerRepository` to avoid reading the entire customer collection when only one customer is needed.
- **State Layer**:
  - Updated `customerByIdProvider` in `CustomersController` to return a `Stream<Customer>` by calling `repository.watchCustomerById(id)` instead of watching the global list and filtering locally. 
  - Regenerated Riverpod files using `build_runner`.
- **UI Layer - Dumb Widgets**:
  - Refactored `OneTimeDepositCard` and `RecurringDepositCard` into pure `StatelessWidget`s (Dumb Widgets), completely decoupling them from Riverpod and data fetching.
  - They now accept explicit `title` and `subtitle` strings instead of a `showCustomerName` flag, enabling true reusability across any context.
- **UI Layer - Smart Composition**:
  - Updated `OneTimeScreen` and `RecurringDepositsScreen` to wrap their card instances in localized `Consumer` widgets. This restricts the scope of rebuilds and Riverpod subscriptions to individual rows in the list view.
  - Updated `CustomerDetailScreen` to directly pass account details to the deposit cards since the customer context is already known.
  - Fixed compilation errors in `CustomerSelectionField`, `OneTimeDepositDetailScreen`, and `RecurringDepositDetailScreen` by properly handling the `AsyncValue<Customer>` state now that the provider returns a Stream.

## Next Steps
- Implement single-document fetching for `OneTimeDepositRepository` and `RecurringDepositRepository`.
- Complete the transition of detail screens from list-based filtering to single-document fetching (`AsyncValue<T>`) and rewrite `AsyncEntityBuilder` accordingly.
